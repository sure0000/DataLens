# 企业语义层与域内自治实践

> 整理自 DataLens 项目讨论 · 2026-05-25  
> 主题：AI 时代本体/语义层的意义、行业实践、DataLens 落地方式、存储结构、联邦治理 vs 企业统一语义层

---

## 摘要

在 AI 时代，语义层的核心价值已从「学术本体表示」转向**企业 AI 的语义地基**——让大模型理解业务概念、可遍历的关系、可执行边界与治理规则。  
对大多数企业而言，**优于「统一编写全部语义内容」的模式，是「域内自治 + 企业薄层（规范/互联/治理）+ 通用架构与工具」**。  
在小型业务域中，**把真实数据与指标口径描述清楚**，往往比建设 formal 本体更划算；DataLens 正是这一思路的工程化实践。

---

## 一、本体建模在 AI 时代有什么意义？

大模型擅长「会说话」，但企业真正需要的是**在特定业务语境里说得对、查得准、还能按规则行动**。本体（Ontology）的角色，已从早年偏学术的 RDF/OWL，演化为**企业语义层 / 决策图谱**——给 AI 一套可治理的「业务语言」。

### 1.1 为什么现在又变得重要？

| 痛点 | 纯 RAG / 纯 LLM | 语义层能补什么 |
|------|-----------------|----------------|
| 同名异义（「客户」在不同系统含义不同） | 容易混用概念 | 统一企业词汇表 |
| 多跳关系推理（供应商→物料→产线→质量） | 向量检索常丢结构 | 显式关系 + 图遍历 |
| 幻觉与不可追溯 | 黑盒生成 | 约束推理 + 血缘引用 |
| Agent 只会答、不会干 | 缺业务动作模型 | 对象 + 关系 + **动作/规则** |
| 跨系统协作 | 各说各话 | 跨域共享语义 |

**一句话**：LLM 提供语言能力，语义层提供业务意义与边界——模型负责理解与生成，语义层负责**锚定、约束、串联、治理**。

### 1.2 行业主流路线（2024–2026）

| 厂商 | 产品/方向 | 侧重点 |
|------|-----------|--------|
| **Palantir** | Foundry Ontology + OSDK | 最深、最偏运营：对象/链接/动作/权限一体的数字孪生 |
| **Microsoft** | Fabric IQ Ontology (Preview) | OneLake 上的实体—关系语义层，服务 Copilot/Agent |
| **Google** | Knowledge Catalog / Dataplex 演进 | Agent 的「上下文引擎」，高精度语义检索 |
| **Databricks + Neo4j** | OntoBricks、GraphRAG | Lakehouse + OWL/图推理 + Agent 部署 |
| **国内** | 用友 LOM、悦点 Knora/SCOPA 等 | 领域本体 + Agent 编排，从理解到执行 |

### 1.3 效果如何？（分场景）

**做得好的：**

- Palantir 供应链等复杂运营场景有公开 ROI 案例（多 ERP 集成、采购优化分钟级、pipeline 归因等）
- GraphRAG / 本体增强 RAG 在多跳问答、根因追溯等场景，准确性与可解释性通常优于纯向量 RAG
- 本体把「能查什么、能改什么、谁能改」写进模型，适合受监管行业

**局限与失败模式：**

- 建设与维护成本高（ontology drift）
- 不是万能降幻觉药——数据绑定差、规则不全照样「看起来很美、用起来很假」
- 轻量 ChatBI 场景，精心做的 schema + RAG 往往 ROI 更高
- 厂商成熟度参差：Palantir 贵且重；Microsoft/Google 仍偏早期

**总体判断**：高价值、高复杂、强治理场景值得投入；轻量问答场景 unified ontology 通常不划算。

## 二、小型业务域的关键：把真实数据描述清楚

在细粒度业务域中，**不必先上 formal 本体**。只要能把真实数据的结构、分布、业务含义、口径与用法说清楚，Copilot/Agent 效果往往已经很好。

DataLens 采用**四层叠加**的轻量语义路线（非 OWL 三元组库）：

```
物理层（Schema/DDL/样本）
    ↓
统计层（Profiler 画像）
    ↓
语义层（LLM 列/表理解）
    ↓
业务层（域描述 + 知识库 + 代码分析）
    ↓
Copilot Prompt 拼装
```

### 2.1 第一层：物理事实（连库读取）

- 列名、类型、库内 COMMENT、DDL、行数、样本（最多 1000 行）
- 写入 `tables`、`columns` 表

### 2.2 第二层：统计画像（Profiler）

- 空值率、去重、Top 值、数值分位数、枚举检测（含 MySQL ENUM 解析）
- 写入 `columns.null_ratio`、`sample_values`、`top_values`、`quality_metrics`（JSON）

### 2.3 第三层：LLM 语义

**列级** → `semantic_desc`、`semantic_type`（metric/dimension/time/id/enum）、`is_usable`、`aggregation_hint`

**表级** → `TableSummary` 五段式长文：
1. 业务描述（含行粒度）
2. 数据定位
3. 核心口径
4. 使用建议
5. 风险边界

另含 `use_cases`、`key_columns`、`warnings`（`|` 分隔存储）

分析时注入**业务域上下文**：域名称、域描述、相关知识条目（按表名/列名相关性排序）。

### 2.4 第四层：人工与外部知识

| 机制 | 作用 |
|------|------|
| **业务域** | 域描述 + 域绑库表 + 域绑知识库 |
| **知识库条目** | Markdown 文档，semantic_role 分类（指标/列词典/JOIN 指南等） |
| **表—知识固定链接** | `TableKnowledgeEntry`：Copilot 固定全文注入 |
| **代码库分析** | 从 Git SQL/ORM 提取枚举值、聚合方式、表引用 |
| **历史问答** | QueryExample + 向量 Few-shot |

### 2.5 Copilot 如何使用这些描述

`SqlCopilotContext` 四层拼装：

1. **BUSINESS CONTEXT — 知识库与业务口径**
2. **BUSINESS CONTEXT — 数据源与表分析**
3. **BUSINESS CONTEXT — 结构化字段**（列级 schema 行）
4. **FEW-SHOT — 历史相似问答**
5. **TABLE SUMMARY — 表级摘要聚合**

## 三、描述信息的存储结构（DataLens）

全部存在 **PostgreSQL + pgvector**，关系表为主、JSON 为辅、向量为检索补充。**没有**单独的 ontology 文件或 RDF 三元组库。

### 3.1 表/列语义核心

**`tables`（TableMeta）**

| 字段 | 含义 |
|------|------|
| table_name, database_name, source_type | 物理定位 |
| datasource_id | 关联数据源 |
| ddl, row_count | 建表语句、行数 |
| status | pending / analyzing / done / error |

**`columns`（ColumnMeta）**

| 类别 | 字段 |
|------|------|
| 物理 | column_name, data_type, comment |
| LLM 语义 | semantic_desc, semantic_type, is_usable, usable_reason |
| 统计 | null_ratio, distinct_count, sample_values, top_values |
| 扩展 | quality_metrics (JSON) |

**`quality_metrics` JSON 典型结构：**

```json
{
  "risk_level": "low",
  "distribution": { "min": 0, "max": 9999, "p25": 10, "p50": 50, "p75": 200 },
  "enum": {
    "kind": "mysql_enum | low_cardinality_observed | code_observed",
    "values": ["paid", "pending"],
    "note": "取值来源说明"
  },
  "aggregation_hint": "sum"
}
```

**`table_summary`（TableSummary）**

- summary（五段式长文）
- use_cases、key_columns（`|` 拼接）
- warnings
- generated_at（支持多版本，Copilot 取最新）

### 3.2 业务域

| 表 | 作用 |
|----|------|
| business_domains | 域实体 |
| business_domain_descriptions | 域说明（可多条，取最新） |
| business_domain_selections | 域 ↔ 数据源/库/表（table_name NULL = 整库） |
| business_domain_knowledge_bases | 域 ↔ 知识库（M:N） |

### 3.3 知识库（双轨）

**传统条目 `knowledge_entries`**

- title, summary, body
- semantic_role, tags, source_meta (JSON)
- source_meta 可含 `pending_table_refs`（代码分析暂存，表登记后 catch-up）

**文档流水线 `documents` + `document_chunks`**

- 大文档分块 + chunk 级 embedding + tsvector 全文检索
- **`document_chunks.semantic_meta`**（JSON）：`semantic_role`、`grounding`（table_refs/column_refs）、`join_edges`、`confidence`
- 流水线 Stage 5 语义结构化（需 LLM；无 LLM 时跳过，不影响索引）

**关联表**

- table_knowledge_bases：表 ↔ 知识库
- table_knowledge_entries：表 ↔ 条目（固定全文注入）

### 3.4 结构化语义抽取（已实现）

| 表 | 内容 |
|----|------|
| business_terms | 业务术语；`concept_id`、related_fields、审核 status |
| metric_definitions | 指标公式、口径、`bound_table_refs`、`concept_id` |
| data_lineage | 表间血缘（Git LLM 提取 + join_guide 同步） |
| **semantic_relations** | 轻量关系图：`term_column` / `metric_table` / `table_join` / `concept_alias` |
| pipeline_runs | 语义提取流水线执行状态 |

关系同步入口：`semantic_relation_sync.py`（文档结构化后 + 语义提取 Step 4）。

### 3.5 向量索引 `embeddings`

| ref_type | 用途 |
|----------|------|
| column | 列语义 + 样本 |
| table | 表摘要 |
| query | 历史问答 Few-shot |
| knowledge_entry | 知识条目分块 |

另：`document_chunks.embedding` 服务新文档流水线混合检索。

### 3.6 存储形态小结

| 描述类型 | 存储形态 |
|----------|----------|
| 物理结构 | 关系列 |
| 统计画像 | 关系列 + quality_metrics JSON |
| 列/表 AI 语义 | Text + 枚举字符串 |
| 业务域口径 | 独立 Text + M:N 关联 |
| 文档知识 | Text + source_meta JSON |
| 代码推导 | JSON 嵌套于 quality_metrics |
| 语义关系 | semantic_relations 关系表 + concept_id |
| 检索 | pgvector + tsvector hybrid |

## 四、联邦语义 vs 企业统一语义层

### 4.1 核心结论

**各细粒度团队维护本域数据与指标口径，必要时维护域间关系——优于「企业统一编写全部语义内容」。**

更准确的模式是：

> **域内厚（内容） + 企业薄（架构/治理/互联） + 通用工具**

不是二选一，而是分工：

```text
企业层（薄）
  ├── 共享词汇与标识规范
  ├── 跨域关系注册 / 血缘
  ├── 治理：权限 · 审计 · 版本
  └── 通用语义架构 + 工具链

域 A / 域 B（厚）
  ├── 表/列语义
  ├── 指标口径
  ├── 域内知识库
  └── 必要时声明跨域关系
```

### 4.2 对比

| 维度 | 企业统一语义层 | 域团队自维护 |
|------|----------------|--------------|
| 信息新鲜度 | 易脱节 | 贴近业务 |
| 维护责任 | 中央背锅 | 谁用谁维护 |
| 建设成本 | 前期极重 | 小域可验证 |
| 覆盖深度 | 浅而广 | 深而准 |
| 组织匹配 | 与 Conway 定律冲突 | 与团队边界一致 |

统一语义层失败常见模式：建 1–2 年「企业本体」→ 业务不认 → 无人更新 → 没人敢用。

### 4.3 通用语义架构应包含什么

| 语义对象 | 域内维护 | 跨域时企业协调 |
|----------|----------|----------------|
| 物理绑定 | 表/列 ↔ 数据源 | 命名规范 |
| 列/表语义 | desc、类型、枚举、聚合 | 同义概念映射 |
| 指标口径 | 公式、过滤、版本 | 公司级指标对齐 |
| 业务规则 | 默认过滤、风险边界 | 合规统一策略 |
| 关系 | 域内 JOIN | 域间实体对齐 |
| 血缘 | 域内流水线 | 跨域依赖 |
| 知识叙述 | 文档、代码、问答 | 可选：企业术语表 |

**工具**要解决：接入 → 画像 → 补语义 → 绑定知识 → 发布给 AI → 版本审核——而不是让大家填三年 Ontology 编辑器。

### 4.4 DataLens 与联邦模型的对应

| 能力 | 谁维护 | 跨域 |
|------|--------|------|
| 域内表/列语义 | 域团队 | — |
| 域内知识/指标 | 域团队 | — |
| 域 ↔ 表范围 | 域团队 | — |
| 代码分析补枚举/聚合 | 自动 + 域团队 | — |
| 跨表关系 | DataLineage（弱） | **待加强** |
| 企业统一词汇 | **尚未做强** | 共享术语表、跨域映射 |

DataLens 已在实践「域内厚、工具通用」；企业推广应补**跨域契约**（同义词、关系注册、口径版本），而非把所有域塞进一个 ontology。

## 五、实践建议

### 5.1 不要这样立项

- ❌ 「三年建成企业统一语义层」
- ❌ 追求 ontology 覆盖率 100%

### 5.2 宜这样立项

- ✅ 语义架构标准 + 域内自助工具 + 跨域关系登记机制
- ✅ 每个域 MVP：几张核心表 + 指标口径条目 + 域描述 + 必要 JOIN/血缘

### 5.3 成功标志

- 域内 Copilot/SQL 准确率可接受
- 口径变更有 owner、有版本
- 跨域问题能查到「谁定义、和谁对齐、关系在哪」

### 5.4 域团队最小维护清单（DataLens 视角）

1. 创建**业务域**，写域描述
2. **挂载**数据源/库/核心表（BusinessDomainSelection）
3. 对核心表跑**分析任务**（ColumnMeta + TableSummary）
4. 建立**知识库**：指标口径、列词典、易错 JOIN 说明
5. 可选：Git 同步代码库，触发**代码分析**补枚举/聚合
6. 跨域时：在知识库或 DataLineage 中**登记关系**，与企业术语表对齐

### 5.5 企业薄层最小规范（建议）

| 字段/对象 | 说明 |
|-----------|------|
| concept_id | 跨域统一概念标识（如 `customer.account_id`） |
| aliases | 各域别名列表 |
| owner_team | 口径负责团队 |
| version / effective_date | 口径版本 |
| relates_to | 跨域关系（source → target, join_key） |
| status | draft / approved / deprecated |

## 六、关键决策记录

| 决策 |  rationale |
|------|------------|
| 小型域优先「描述清楚真实数据」而非 formal OWL | ROI 高、维护成本低、与 LLM+RAG 链路匹配 |
| 语义内容域内自治 | 新鲜度、责任清晰、符合组织边界 |
| 企业提供架构+工具+治理薄层 | 避免大一统本体项目失败 |
| DataLens 用 PostgreSQL 关系表+JSON+向量 | 实用、可演进，非三元组库 |
| Copilot 分层 prompt（知识/表分析/schema/few-shot） | 业务口径优先于裸 schema |
| 知识库清洗走轻量本体（semantic_meta + semantic_relations） | 非 formal OWL，域内厚+企业薄 concept_id |
