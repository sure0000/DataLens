# 本体建模理论标准

> 本文档定义企业级本体建模的理论基线，覆盖 W3C 语义网技术栈、标准方法论、规范语言分工及反模式。作为 DataLens 本体建模能力的设计参照，非对外产品文档。

---

## 一、W3C 语义网技术栈

本体建模依赖 W3C 定义的分层技术栈（Semantic Web Layer Cake）。以下按自底向上列出与 DataLens 相关的各层：

| 层 | 标准 | 角色 |
|----|------|------|
| 标识符 | IRI / URI | 资源唯一标识 |
| 数据模型 | RDF 1.1 | 三元组（主语-谓词-宾语）图模型 |
| Schema | RDFS | 类/属性层级、domain/range 基础约束 |
| 分类法 | SKOS | 轻量概念体系：prefLabel、broader/narrower、altLabel |
| 本体语言 | OWL 2 | 描述逻辑：类表达式、属性特征、公理推理 |
| 校验 | SHACL | 封闭世界数据校验：形状（Shape）约束 RDF 图 |
| 查询 | SPARQL 1.1 | RDF 图遍历与检索 |
| 规则 | SWRL / RIF | Horn 子句规则扩展 OWL 表达力 |

**关键关系：**
- RDF 是一切的上层数据模型
- RDFS + OWL = 推理层（TBox 模式 + ABox 实例）
- SKOS = 轻量组织层（无需描述逻辑推理）
- SHACL = 校验层（与 OWL 互补，非替代）
- SPARQL = 消费层（查询已推理或已校验的图）

## 二、本体工程的经典方法论

### 2.1 Noy & McGuinness「Ontology Development 101」

Stanford KSL 提出的七步迭代法，是本体工程领域引用最广的入门方法论：

| Step | 活动 | 产出物 |
|------|------|--------|
| 1 | 确定领域与范围 | 领域定义 + 能力问题（Competency Questions） |
| 2 | 考虑复用已有本体 | 复用评估（FOAF、Dublin Core、PROV-O 等） |
| 3 | 枚举重要术语 | 未结构化的术语清单 |
| 4 | 定义类与类层次 | `rdfs:subClassOf` 层级（is-a 关系） |
| 5 | 定义属性 | ObjectProperty（对象间关系）+ DatatypeProperty（字面量属性） |
| 6 | 定义约束 | 基数（cardinality）、值类型、domain/range |
| 7 | 创建实例 | ABox 个体填充 |

**核心原则：**
- 没有唯一正确的建模方式——取决于应用场景
- 本体开发是迭代过程，非一次性交付
- 类 = 名词（领域对象），属性 = 动词（领域关系）

### 2.2 OntoClean 方法论（Guarino & Welty）

在 Noy & McGuinness 基础上补充形式化标准：

| 元属性 | 含义 | 检查规则 |
|--------|------|----------|
| Rigidity（刚性） | 实例是否必然属于该类 | 刚性类（Person）不能包含反刚性类（Student）作为子类 |
| Identity（同一性） | 如何判断两个实例相同 | 同类必须有相同的 identity criteria |
| Unity（整体性） | 实例是否有明确的整体边界 | 整体类（Table）vs 可散类（Water） |

**对 DataLens 的启示：** BusinessTerm、Metric、Dimension 应当是 owl:disjointWith 的（当前已实现），因为它们有不同的 identity criteria。

### 2.3 企业本体的四层架构

业界（包括 OpenSPG、OntoKG）普遍采用的分层架构：

```
Layer 0: Enterprise Ontology  →  跨域共享的顶层概念（如「客户」「订单」的通用定义）
Layer 1: Domain Ontology      →  各业务域的 TBox 扩展（如「交易域-退款单」「营销域-优惠券」）
Layer 2: Application Ontology →  知识库/数据源的 ABox 实例（如具体的表、指标、术语）
Layer 3: Inference Closure    →  推理闭包（subClassOf 传递、property chain 物化）
```

DataLens 当前 TBox 已遵循此架构（`core.ttl` = Layer 0/1, `kb/{id}` = Layer 2, `inferred/{scope}` = Layer 3）。

## 三、TBox / ABox / RBox 三元结构

### 3.1 定义

| 组件 | 全称 | 内容 | 对应 OWL 元素 |
|------|------|------|--------------|
| TBox | Terminological Box | 模式层：类定义、属性定义、公理 | `owl:Class`、`rdfs:subClassOf`、`owl:disjointWith`、`owl:Restriction` |
| ABox | Assertional Box | 实例层：个体断言 | `rdf:type`、ObjectProperty 断言、DataProperty 断言 |
| RBox | Role Box | 属性特征层：属性的数学特性 | `owl:TransitiveProperty`、`owl:SymmetricProperty`、`owl:FunctionalProperty`、`owl:inverseOf`、`owl:propertyChainAxiom` |

### 3.2 严格分离原则

- TBox 变更应通过版本管理（`owl:versionInfo`、`owl:deprecated`）
- ABox 写入应经过 SHACL 校验（写入前验证）
- RBox 应与 TBox 同步发布，推理引擎依此工作

## 四、OWL 2 语言家族与 Profile 选择

### 4.1 三个 Profile 的取舍

| Profile | 计算复杂度 | 支持 | 不支持 | 适用场景 |
|---------|-----------|------|--------|----------|
| OWL 2 EL | PTime（多项式） | 大规模分类、医疗本体（SNOMED） | `owl:unionOf`、`owl:complementOf`、cardinality > 0 | 需要高效 subClassOf 推理 |
| OWL 2 QL | NLogSpace | 合取查询重写、数据库映射 | `owl:unionOf`、cardinality、FunctionalProperty | 本体作为数据库查询接口 |
| OWL 2 RL | PTime | 规则式推理、RDF 图扩展 | `owl:minCardinality`、qualified cardinality、`owl:unionOf` | **企业知识图谱**（DataLens 的选择） |
| OWL 2 DL | NExpTime | 完整描述逻辑 | — | 需要完整表达力的学术场景 |

### 4.2 DataLens 选择的 RL Profile 约束

RL profile 核心限制：

- **不支持**任何 `owl:minCardinality`、`owl:cardinality`（任何值 > 0）
- **不支持**qualified cardinality（`owl:minQualifiedCardinality` 等）
- **仅支持** `owl:maxCardinality 0` 和 `owl:maxCardinality 1`（且仅在超类位）
- **不支持** `owl:unionOf`、`owl:complementOf`、`owl:oneOf`
- **支持** `rdfs:subClassOf`、`owl:disjointWith`、`owl:TransitiveProperty`、`owl:SymmetricProperty`、`owl:FunctionalProperty`、`owl:inverseOf`、`owl:propertyChainAxiom`

这意味着：**「每个订单必须关联一个客户」这类约束无法在 OWL 2 RL 中表达**，必须交给 SHACL。

### 4.3 `owl:Restriction` 的四种基本形式

即使 RL Profile 受限，以下存在量词仍被部分支持：

```turtle
# Existential — RL 支持
ex:Order  rdfs:subClassOf  [ owl:onProperty ex:hasCustomer ; owl:someValuesFrom ex:Customer ] .

# Universal — RL 限制使用
ex:Order  rdfs:subClassOf  [ owl:onProperty ex:hasItem ; owl:allValuesFrom ex:Product ] .
```

## 五、OWL vs SHACL：最常被混淆的两个标准

### 5.1 核心区别

| 维度 | OWL（推理语言） | SHACL（校验语言） |
|------|---------------|-----------------|
| 假设 | **开放世界**（缺失不等于假） | **封闭世界**（缺失 = 违规） |
| 目的 | 推导新知识、检查逻辑一致性 | 验证数据完整性、合规性 |
| 处理缺失 | 不做任何假设 | 报告违规 |
| 处理冲突 | 推断等价（如两值必为同一实体） | 报告违规 |
| 典型用例 | 分类推理、属性链传递 | 数据质量检查、表单校验 |
| 输出 | 推导出的新三元组 | 校验报告（conforms + 违规详情） |

### 5.2 正确分工

```
OWL 负责：              SHACL 负责：
  ├── 类层级推理          ├── 必填字段校验
  ├── 属性传递闭合        ├── 数据类型校验
  ├── 同义实体合并        ├── 基数约束（"必须有 X"）
  ├── 概念分类            ├── 值域合规检查
  └── 属性链推导          └── 跨实体引用完整性
```

### 5.3 常见反模式

**反模式 1：用 OWL 做数据校验**
「我定义了 `owl:minCardinality 1`，为什么数据没有客户时不报错？」
→ 因为 OWL 不是校验语言。这个需求应该用 `sh:minCount 1`。

**反模式 2：用 SHACL 做推理**
「我定义了 Shape 约束，为什么系统不自动推导出这个人的类型？」
→ 因为 SHACL 不做推理。这个需求应该用 `rdfs:subClassOf` + OWL 推理机。

**反模式 3：把 OWL cardinality 误读为「强制约束」**
文档中常见的错误：「owl:cardinality 约束表示每个订单必须有客户」
→ 在开放世界下，`owl:cardinality 1` 的含义是：「如果一个个体有 hasCustomer，那它最多只有一个」。它不保证「一定有」，只保证「不会多于 1 个」。

## 六、SKOS：轻量级知识组织

SKOS 定位介于自由文本和形式本体之间：

| SKOS 元素 | 用途 | OWL 等价 |
|-----------|------|----------|
| `skos:Concept` | 概念节点 | `owl:Class` 的实例 |
| `skos:prefLabel` | 首选标签 | — |
| `skos:altLabel` | 替代标签（同义词） | — |
| `skos:definition` | 定义 | — |
| `skos:broader` / `skos:narrower` | 层级关系 | `rdfs:subClassOf`（但语义更弱） |
| `skos:related` | 关联关系 | — |
| `skos:exactMatch` | 跨体系等价 | `owl:equivalentClass` |
| `skos:inScheme` | 归属概念体系 | — |

**使用原则：** 术语组织和轻量分类用 SKOS；需要推理、约束、复杂公理时用 OWL。DataLens 正确地混用了两者——术语的标签/同义词用 SKOS，表/指标/血缘关系用 OWL 属性。

## 七、SWRL：规则扩展的边界

### 7.1 SWRL 能做什么

SWRL 以 Horn 子句扩展 OWL：

```
前提（Body） → 结论（Head）
```

合法示例：
```
Person(?x) ∧ hasAge(?x, ?age) ∧ swrlb:greaterThan(?age, 18) → Adult(?x)
```

DataLens 场景示例：
```
Metric(?m) ∧ computedFromTable(?m, ?t) ∧ hasColumn(?t, ?c) → mapsToColumn(?m, ?c)
```

### 7.2 可判定性约束

- 完整 SWRL 是**不可判定的**
- 只有 **DL-Safe SWRL**（规则变量仅绑定已知个体）是可判定的
- 多数推理机对 SWRL 的支持是有限且性能敏感的

### 7.3 实际建议

| 场景 | 推荐方案 |
|------|----------|
| 属性传递、对称性 | OWL property characteristics（∅ SWRL） |
| 简单条件推导 | DL-Safe SWRL 或 Datalog 规则 |
| 复杂业务流程 | 应用层规则引擎，不入本体 |
| 模糊匹配、概率推理 | 外部服务，不入本体 |

## 八、DQV：数据质量词汇表（W3C）

### 8.1 标准五维度

| 维度 | 定义 | 计算公式 |
|------|------|----------|
| 完整度 Completeness | 有完整定义的实体占比 | 有定义实体 / 总实体 |
| 准确度 Accuracy | 通过校验的三元组占比 | SHACL 通过三元组 / 总三元组 |
| 一致性 Consistency | 无冲突实体占比 | 1 - (冲突实体 / 总实体) |
| 时效性 Timeliness | 近期更新的实体占比 | 90日内更新实体 / 总实体 |
| 唯一性 Uniqueness | 无重复定义实体占比 | 去重后实体 / 去重前实体 |

### 8.2 RDF 表示

```turtle
ex:CompletenessMeasurement
  a dqv:QualityMeasurement ;
  dqv:isMeasurementOf dqv:CompletenessMetric ;
  dqv:inDimension dqv:CompletenessDimension ;
  dqv:value "0.83"^^xsd:decimal ;
  dqv:computedOn ex:DomainEntity .
```

DataLens 的 `governance.ttl` 已正确定义了这 5 个 DQV 维度和指标。

## 九、本体建模的标准七类产出物（修正版）

对照 Noy & McGuinness 方法论和 W3C 技术栈，以下是企业本体建模的标准产出物。**与常见科普文档的区别用粗体标出。**

### 1. 概念词典 / 业务词汇表

- **技术基座：** SKOS
- **产出格式：** 表格（Excel/Markdown）+ RDF（`skos:Concept` + `skos:prefLabel` + `skos:definition`）
- **包含：** 术语名、类型枚举、业务定义、同义词、上位概念、来源锚点
- 示例：`VIP客户 = 累计消费 > 10000元 且 注册 > 180天`

### 2. 类层次结构

- **技术基座：** RDFS + OWL
- **产出格式：** OWL TBox（`rdfs:subClassOf` 层级）
- **应包含：** `rdfs:subClassOf` 树 + `owl:disjointWith` 公理 + `owl:equivalentClass` 跨本体映射
- **质量要求：** 无循环、兄弟节点同粒度、区分刚性/非刚性类（OntoClean）

### 3. 关系定义（Object Properties）

- **技术基座：** OWL ObjectProperty
- **产出格式：** OWL TBox（属性定义 + domain/range + 特征声明）
- **应包含：** 属性名、domain、range、特征（Transitive / Symmetric / Functional / InverseFunctional）、逆属性、属性链公理
- **注意：** 企业本体通常使用**标准关系类型**而非自由命名的领域谓词，以确保推理和校验的一致性

### 4. 属性定义（Data Properties）

- **技术基座：** OWL DatatypeProperty
- **产出格式：** OWL TBox（属性定义 + `rdfs:domain` + `rdfs:range`（xsd 类型））
- **应包含：** 属性名、domain 类、range 数据类型、`owl:FunctionalProperty` 标记
- **注意：** 数据属性描述**概念的特征值**，不是数据库列定义。与物理表列的映射应通过 `mapsToColumn` 单独维护

### 5. 公理与规则约束（**关键修正**）

**这一层应严格拆分为三种独立技术：**

#### 5a. OWL 推理公理（开放世界，用于推理）

| 公理类型 | 示例 | DataLens RL 支持 |
|----------|------|------------------|
| `rdfs:subClassOf` | Manager ⊑ Employee | ✅ |
| `owl:disjointWith` | Metric disjointWith Dimension | ✅ |
| `owl:FunctionalProperty` | platformId is functional | ✅ |
| `owl:TransitiveProperty` | transformsFrom is transitive | ✅ |
| `owl:SymmetricProperty` | joinableWith is symmetric | ✅ |
| `owl:inverseOf` | hasChunk inverseOf partOf | ✅ |
| `owl:propertyChainAxiom` | computedFromTable ◦ transformsFrom | ✅ |
| `owl:someValuesFrom` | Order ⊑ ∃hasCustomer.Customer | ⚠️ RL 受限 |
| `owl:allValuesFrom` | Order ⊑ ∀hasItem.Product | ⚠️ RL 受限 |

#### 5b. SHACL 校验约束（封闭世界，用于数据质量）

```turtle
ex:OrderShape a sh:NodeShape ;
  sh:targetClass ex:Order ;
  sh:property [
    sh:path ex:hasCustomer ;
    sh:minCount 1 ;              # 必须有客户（与 OWL 开放世界不同！）
    sh:class ex:Customer ;
  ] .
```

#### 5c. SWRL / Datalog 规则（IF-THEN 推导，封闭世界或 DL-Safe）

```
累计消费(?c, ?amt) ∧ swrlb:greaterThan(?amt, 10000) → VIP客户(?c)
```

### 6. 实例数据（ABox）

- **技术基座：** RDF + OWL ABox
- **产出格式：** RDF Turtle / Trig
- **应包含：** rdf:type 声明、ObjectProperty 断言、DatatypeProperty 字面量
- **质量要求：** 所有写入须经 SHACL 校验（通过 → 生产图，失败 → 隔离区）

### 7. 知识图谱可视化

- **不是** W3C 标准的一部分，而是消费层的工程产出
- 最小可用：D3.js / vis.js 力导向图
- 企业级应支持：层级视图、过滤/搜索、路径高亮、血缘追溯、导出

## 十、常见误区与反模式

| 误区 | 正确理解 |
|------|----------|
| 「OWL 公理 = 业务硬约束」 | OWL 公理是**推理规则**（开放世界），硬约束需用 SHACL（封闭世界） |
| 「Cardinality 保证数据完整性」 | `owl:minCardinality 1` ≠ 必须有值，SHACL `sh:minCount 1` 才保证必须有值 |
| 「SWRL 可以随便写规则」 | SWRL 不可判定，应限制为 DL-Safe 或改用 Datalog |
| 「可视化是可选的」 | 对于企业采纳，可视化是**最重要的沟通工具**，不应省略 |
| 「概念词典是初级产出物」 | 概念词典是**价值最高的产出物**，解决了 80% 的语义对齐问题 |
| 「本体建模就是写 RDF」 | 本体建模的核心是**概念分析和领域建模**，RDF 只是序列化格式 |

## 九附录：DataLens 抽取流水线与七类产出物对应关系

| 标准产出物 | DataLens 抽取步骤 |
|-----------|------------------|
| ① 概念词典 | 术语抽取（term_extraction） |
| ② 类层次结构 | 层级构建（hierarchy_building） |
| ③ 关系定义 | 关系抽取 + 血缘抽取 + Join 抽取 |
| ④ 属性定义 | 属性值提取（嵌入各实体抽取步骤） |
| ⑤a OWL 推理公理 | 规则抽取（ruleType=owl_axiom）+ TBox 手写公理 |
| ⑤b SHACL 校验约束 | 规则抽取（ruleType=shacl_constraint）+ 手写 Shape |
| ⑤c SWRL 规则 | 规则抽取（ruleType=swrl_rule）+ 手写推理规则 |
| ⑥ ABox 实例 | 全部实体抽取步骤的最终产出 |
| ⑦ 对齐与可视化 | 跨库等价检测 + 跨域本体对齐（后处理） |
