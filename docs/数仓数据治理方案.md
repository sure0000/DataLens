# 数仓数据治理方案

---

## 第一部分：数仓数据治理问题汇总

### 1. 指标口径不一致

| 表现 | 示例 |
|---|---|
| **同名不同义** | 都叫"活跃用户"，A部门按登录、B部门按交易、C部门按页面访问 |
| **同义不同名** | 同样计算"去重购买用户"，有的叫"买家数"、有的叫"支付用户数"、有的叫"成交UV" |
| **口径边界模糊** | "近7天"有的含当天、有的不含；"有效订单"有的排除退款、有的不排除 |
| **计算逻辑隐蔽** | 指标名称背后是几百行SQL，消费方只知道名称不知道逻辑，出了问题无人能解释 |
| **多源定义冲突** | 同一指标在PRD文档、BI报表、数据仓库ETL脚本中定义不同，不知道以哪个为准 |

**根本原因**：缺乏统一的指标定义平台，指标定义散落在各个报表、SQL脚本、Excel、文档中。

### 2. 数据质量问题（六维）

| 维度 | 问题 | 示例 |
|---|---|---|
| **完整性** | 空值、缺失记录、断链数据 | `order.user_id` 30% 为空 |
| **准确性** | 数据与真实业务不符 | `total_amount` 出现负数 |
| **一致性** | 同一数据在不同表/系统中值不同 | `ods.users` 和 `dwd.users` 同一用户状态不同 |
| **时效性** | 数据更新不及时 | ETL失败导致表数据停留在一周前 |
| **唯一性** | 重复数据 | 同一订单出现多条记录 |
| **有效性** | 格式错误、枚举越界、类型不匹配 | `status` 字段出现不存在的枚举值 |

**根本原因**：缺乏周期性的数据质量检查和监控，问题只能在被人发现时被动修复。

### 3. 数据链路不透明

- 数据从源表 → ETL中间表 → 汇总表 → 指标 → 报表的链路不可见
- ETL过程的中间表、临时表文档缺失
- 下游出问题时无法快速追溯上游根因
- "这个指标用到了哪些源表？这张表影响了哪些指标？"回答不了

### 4. 变更影响不可控

- 上游表结构变更（加列、删列、改类型）→ 下游指标悄悄出错
- ETL逻辑调整 → 历史数据口径变化 → 报表数据"漂移"
- 缺乏变更通知和影响分析机制

## 第二部分：业界通用解决方案

### 1. 指标管理平台（Metric Platform / Headless BI）

**代表**：Airbnb Minerva、Uber uMetric、阿里 DataWorks、网易有数、dbt MetricFlow

**核心思路**：指标 = 名称 + 口径描述 + 计算公式 + 维度 + 标签，指标作为一等公民统一管理，所有下游消费（报表、API、分析）都从此消费。

**关键能力**：
- **指标目录**（Metric Catalog）：注册、搜索、发现指标
- **指标血缘**：指标 ← 计算逻辑 ← 源表/字段
- **指标认证**：certified（官方认证）vs. community（社区贡献）
- **指标比对**：同名指标口径diff，冲突检测

### 2. 数据质量平台（Data Quality Platform）

**代表**：Great Expectations、Soda、Monte Carlo、dbt tests

**核心思路**：对数据定义质量约束（expectation），周期性验证，失败时告警。

**关键能力**：
- **规则定义**：声明式规则配置，自动画像建议
- **调度执行**：定期运行，支持增量/全量
- **结果管理**：趋势追踪、SLA、历史基线对比
- **告警与事件**：通知 → 升级 → 关闭的事件管理闭环

### 3. 数据血缘（Data Lineage）

**代表**：DataHub、OpenLineage、Atlas、Marquez、dbt docs

**核心思路**：追踪从数据源头到消费终端的完整链路。

**关键能力**：
- **表级血缘**：源表 → ETL中间表 → 汇总表
- **字段级血缘**：某字段 = f(上游字段A, 上游字段B)
- **可视化**：DAG 图 + 点击下钻
- **影响分析**：改了上游，影响哪些下游

### 4. 数据目录（Data Catalog）

**代表**：Alation、Collibra、Atlan、Amundsen

**核心思路**：企业数据资产的黄页，让用户能发现、理解、信任数据。

**关键能力**：搜索与发现、业务词汇表、分类与标签、所有权管理、热度与使用统计

## 第三部分：基于 DataLens 的方案设计

### 为什么不做通用平台

DataLens 的本质是一个 **ChatBI 系统**，不是数据治理中台。它不需要：
- 做全量的数据质量规则管理（不是 Great Expectations）
- 做通用的数据目录（不是 DataHub）
- 支持任意数据源的全面质量监控

DataLens 的**独特优势**：
- 已有一套本体知识图谱（RDF），业务概念、指标、维度、关系已经建模好了
- 已有 LLM 驱动的内容理解能力（语义提取、口径比对）
- 已有 SHACL 校验框架（可扩展到数据层）
- 已有 APScheduler + SQL 执行能力
- 已有列画像能力（profiler）
- **本体是 DataLens 做数据治理的锚点**——可以基于语义来做治理，而非基于物理表名

### 方案总览

三个核心方案，围绕 DataLens 的**本体图**展开：

```
                    ┌───────────────────────────┐
                    │   本体知识图谱 (RDF)        │
                    │  概念 · 指标 · 维度 · 关系  │
                    └───────────┬───────────────┘
                                │
          ┌─────────────────────┼─────────────────────┐
          │                     │                     │
          ▼                     ▼                     ▼
  ┌───────────────┐   ┌───────────────┐   ┌───────────────┐
  │ 方案一：       │   │ 方案二：       │   │ 方案三：       │
  │ 指标本体治理   │   │ 语义级质量画像  │   │ 本体影响图     │
  │               │   │               │   │               │
  │ 解决：指标口径 │   │ 解决：表级/字段 │   │ 解决：链路不透明│
  │ 不一致        │   │ 级数据质量问题  │   │ 变更影响不可控  │
  └───────────────┘   └───────────────┘   └───────────────┘
```

### 方案一：指标本体治理（解决「指标口径不一致」）

#### 核心思路

DataLens 已有 `dl:Metric` 实体存储在 RDF 中，指标通过 LLM 从文档/代码中提取。问题是：**同一指标名从不同文档提取出了不同的定义，系统没有发现和解决这种冲突**。

#### 具体设计

**1.1 指标注册与聚合**

按指标名称（`prefLabel`）聚合所有来源的指标定义：
- RDF `dl:Metric`（来自文档提取）
- `KnowledgeEntry` with `semantic_role = 'business_metric'`（来自知识库导入）
- `DocumentChunk` 中包含指标定义的片段

聚合后得到：`{ 指标名 → [定义来源1, 定义来源2, ...] }`

**1.2 口径一致性比对**

对同一指标名的多个定义，两两比对：

```
步骤1: 规则初筛
  - formula 文本完全相同 → "identical"
  - 口径文本完全相同 → "identical"  
  - 来源相同（同一文档）→ 跳过（same source）

步骤2: LLM 语义比对（规则筛不掉的对）
  Prompt: 对比指标"{name}"的两个定义：
    定义A (来源{a.source})：口径={a.caliber}，公式={a.formula}
    定义B (来源{b.source})：口径={b.caliber}，公式={b.formula}
    
    判断冲突类型：
    - identical: 含义完全一致，只是表述不同
    - compatible: 口径兼容，B是A的子集/特例（或反之）
    - conflicting: 口径冲突，关键逻辑不同
    - unrelated: 同名但含义完全不同（不同指标恰好同名）

输出冲突矩阵 + 每个冲突对的分析说明
```

**1.3 指标绑定验证**

验证指标定义中声明的 `computedFromTable` 和 formula 字段：
- `computedFromTable` 指向的表在物理数据源中是否存在
- formula 中引用的字段在对应表的 `ColumnMeta` 中是否存在
- formula 是否可以作为合法的只读 SQL 执行（用 sqlglot 解析 + `execute_readonly_sql` 验证）

验证失败 → 标记为"绑定无效"，指标 confidence 降低。

**1.4 指标一致性评分**

纳入已有的五维 DQV 评分体系（`governance.py`）：
- **一致性维度**：无冲突的指标 / 多源指标总数
- 在知识库质量评估报告中呈现指标一致性子项

**1.5 涉及改动**

| 位置 | 改动 |
|---|---|
| `backend/services/data_quality/caliber.py` | 新增：指标聚合、LLM口径比对、绑定验证 |
| `backend/dq_models.py` | 新增：`DQMetricDefinition`、`DQCaliberComparison` 表 |
| `backend/routers/data_quality.py` | 新增：`/metrics/*`, `/caliber-comparisons/*` API |
| `backend/services/ontology/governance.py` | 改动：metrics一致性分数纳入评估 |
| `frontend/app/data-quality/caliber-comparison/` | 新增：口径比对页面 |
| 已有 `MetricList.tsx`、`MetricDerivationChain.tsx` | 改动：增加冲突标记和绑定验证状态 |

### 方案二：语义级数据质量画像（解决「数据质量问题」）

#### 核心思路

不是建一套独立的 DQ 规则引擎，而是在 DataLens 已有的列画像（profiler）和语义标注上增强：

- **周期性画像**：定期重跑 profiler，存储快照
- **语义级质量规则**：规则绑定到本体概念（如"金额类字段不应为负"），而非物理表名（如 `db.orders.amount >= 0`）
- **本体驱动的影响评估**：某表质量下降 → 降低依赖该表的指标 confidence → Copilot 回答中风险提示

#### 具体设计

**2.1 周期性列画像**

复用已有 `profiler.py:profile_column()`，通过 APScheduler 定期执行：

```
对每个活跃数据源的每张表：
  1. 执行 profile_column() 获取最新画像
  2. 与上次画像对比，计算漂移指标：
     - null_ratio 变化幅度
     - distinct_ratio 变化幅度
     - top_values 分布变化（JS divergence）
     - 数值列 p50 变化幅度
  3. 存储本次画像快照（JSON）
  4. 漂移超过阈值 → 生成质量事件
```

快照存储：在已有 `columns.quality_metrics` (JSON) 基础上，新增一个 `column_quality_snapshots` 表存储历史。

**2.2 语义级质量规则**

与传统 DQ 工具不同，DataLens 的规则应**绑定到语义层**：

```yaml
# 传统方式（绑定物理表）
rule: db.orders.amount >= 0

# DataLens 方式（绑定本体概念）
rule: 
  semantic_type: "order_amount"   # 匹配所有语义标注为"订单金额"的字段
  constraint: ">= 0"
  说明: "所有标注为 'order_amount' 的字段都应 ≥ 0"
```

这样做的好处：
- 一条规则覆盖多个表（跨数据源复用）
- 表结构变化（换库、重命名）不影响规则
- 新增表自动纳入已有规则的检查范围

利用已有的 `ColumnMeta.semantic_type` 和 `ColumnMeta.semantic_desc` 进行规则匹配。

**简化实现**：P0 阶段不实现完整的语义匹配引擎，而是：
1. 扩展现有 profiler 的输出，对每个列自动判断潜在质量问题（null_ratio > 0.3、all values same等）
2. 在表详情页增加"数据质量"Tab，展示最新画像 + 变化趋势
3. 在 Copilot 查询结果中标注所使用的列的质量状态

**2.3 本体影响传播**

```
表数据质量评分下降
  → 查询 SPARQL: "哪些指标 dl:computedFromTable 指向该表？"
  → 自动降低这些指标的 dl:confidence 值
  → Copilot 回答涉及这些指标时，追加质量提示
```

**2.4 涉及改动**

| 位置 | 改动 |
|---|---|
| `backend/services/profiler.py` | 扩展：profile_column 输出增加风险标记列表 |
| `backend/services/data_quality/scheduler.py` | 新增：周期性重跑 profiler |
| `backend/dq_models.py` | 新增：`ColumnQualitySnapshot` 表 |
| `backend/routers/data_quality.py` | 新增：`/dashboard/table-health/{id}`、`/snapshots/*` API |

### 方案三：本体影响图（解决「链路不透明 + 变更影响」）

#### 核心思路

DataLens 的本体图中已经编码了实体间的关系（`dl:computedFromTable`、`dl:derivedFrom`、`dl:dependsOn`、`dl:joinableWith`、`dl:mappedToColumn`），利用 SPARQL 查询就能做影响分析，不需要另外建一套血缘系统。

#### 具体设计

**3.1 影响分析查询**

```
问题："删了表 T，影响哪些业务？"

SPARQL 查询：
  ?metric dl:computedFromTable <table/T> .
  ?metric dl:dependsOn ?concept .
  ?domain dl:contains ?concept .
  
结果：
  影响指标：[M1, M2, M3]
  影响概念：[C1, C2]
  影响业务域：[财务域, 运营域]
```

**3.2 质量事件影响范围**

当数据质量事件触发时，自动计算影响范围：

```
1. 表 X 质量评分从 B 降为 D
2. 查询 SPARQL: 哪些指标依赖表X？
3. 查询 SPARQL: 这些指标属于哪些业务域？
4. 生成影响报告：
   - 直接影响：3个指标
   - 影响业务域：2个
   - 影响 Copilot 问题（通过历史查询匹配）：最近7天有12个相关问题可能受影响
5. 在 UI 中展示影响范围（简单的关系图）
```

**3.3 Copilot 质量感知**

在 Copilot NL-to-SQL 过程中增加质量感知环节：

```
现有流程：
  问题 → 意图识别 → 本体匹配 → SQL生成 → SQL执行 → 返回结果

增加后：
  问题 → 意图识别 → 本体匹配 → SQL生成 → SQL执行 → 返回结果
                                ↓
                         查询关联指标/表的质量评分
                                ↓
                         质量评分 < 阈值 ? 
                           YES → 结果中追加质量提示（"该指标依赖的表X最近数据质量评分较低，建议核实"）
                           NO  → 正常返回
```

利用已有的 `context_builder.py` 和 `routing_bundle.py`，在上下文组装阶段就查询质量信息，注入 prompt 或追加到响应中。

**3.4 涉及改动**

| 位置 | 改动 |
|---|---|
| `backend/services/data_quality/impact.py` | 新增：SPARQL 影响查询函数 |
| `backend/routers/data_quality.py` | 新增：`/impact/table/{id}` API |
| `backend/services/copilot/context_builder.py` | 改动：组装上下文时查询并注入质量信息 |
| `backend/services/copilot/pipeline.py` | 改动：响应中追加质量提示 |
| `frontend/components/data-quality/ImpactView.tsx` | 新增：影响范围展示（简单节点关系图） |

## 第六部分：验证方式

1. **口径比对**：启动 → 导入多份包含指标定义的文档 → 执行口径扫描 → 确认冲突列表显示 → 人工复核 + 标记
2. **列画像周期性更新**：手动触发画像采集 → 确认快照写入 → 等待 cron 自动执行 → 确认趋势数据
3. **仪表盘**：访问 `/data-quality` → 确认汇总数据正确 → 点击下钻
4. **现有功能回归**：Copilot 问答、知识库管理、本体浏览不受影响（新功能为独立模块，不改动核心路径）
